# Hackathon Overview: AI Agents: Intensive Vibe Coding Capstone Project

This document provides a comprehensive structured overview of the Kaggle & Google **AI Agents: Intensive Vibe Coding Capstone Project** hackathon, compiled from all official announcements, requirements, evaluation metrics, and project planning discussions.

## 1. Event Announcement & Core Details

### Overview
The **AI Agents: Intensive Vibe Coding Capstone Project** is the final assignment and culmination of Kaggle’s **5-Day AI Agents: Intensive Vibe Coding Course with Google** (held June 15 - 19, 2026). This hands-on capstone project offers participants the opportunity to apply the knowledge, tools, and best practices gained during the course to solve real-world use cases on Kaggle. 

Participants are tasked with building a practical, real-world AI agent solution that automates tasks, assists users, solves business challenges, or improves everyday workflows.

## 2. Project Tracks & Awards

Judges will review submissions across four distinct tracks designed to inspire focus. The top 3 teams in each track will receive Kaggle swag and have their work featured on Kaggle's social media platforms. Judges reserve the right to move projects between tracks if appropriate.

### Track 1: Agents for Good
* **Focus:** Submissions that help solve problems for humanity and societal challenges.

* **Focus:** Creating agents designed to solve compelling enterprise or business problems where cost, revenue, efficiency, or business impact matters.
* **Examples:** Expense submission management, highlighting pipeline actions, driving automated insights, or creating product innovations.
* **Awards:** * Award 1: Winner will receive Kaggle swag (Non-monetary).

  * Award 3: Winner will receive Kaggle swag (Non-monetary).

### Track 3: Concierge Agents
* **Focus:** Designing safe, useful, and secure personal assistants that streamline and simplify individual, family, or social everyday workflows while keeping user data highly secure.

* **Awards:** * Award 1: Winner will receive Kaggle swag (Non-monetary).

## 3. Submission Requirements

A valid submission must be fully completed and finalized prior to the deadline. Any unsubmitted or draft Writeups at the deadline will not be considered. 

### Mandatory Submission Components:
1. **Kaggle Writeup:** Serves as the official project report.
2. **Media Gallery:** Contains required visual assets.
3. **Attached Public Video:** A YouTube walkthrough link.
4. **Attached Project Link:** A live URL or public repository.

---

### Deep Dive: Submission Components

#### 1. Kaggle Writeup
* **Description:** Your project report containing a title, subtitle, selected Track, and a detailed analysis of your submission. It should showcase the vision, technical implementation, architectural choices, and the project journey.
* **Word Count Limit:** Must not exceed 2,500 words. Submissions over this limit may face penalties.
* *Note on Privacy:* If a private Kaggle Resource is attached to your public Kaggle Writeup, that resource will automatically become public after the submission deadline.

#### 2. Media Gallery & Video
* **Cover Image:** A cover image is strictly required to submit the Writeup.
* **Video Requirements:** A video presentation of **5 minutes or less** must be attached to the Media Gallery and published to **YouTube**. 
* **Video Content Guide:**
  * **Problem Statement:** Explain the core problem and why it is important/interesting to solve.
  * **Why Agents:** Explain why agents are uniquely positioned to solve this specific challenge.
  * **Architecture:** Visual diagrams and text explanations of the overall agent architecture.
  * **Demo:** A live or animated/visual walkthrough of the working solution.
  * **The Build:** Technical details on how it was built, tools used, and technologies integrated.

#### 3. Public Project Link
* **Live Demo:** A URL to the working product or interactive demo so judges can experience it firsthand. It must be publicly accessible with no login walls or paywalls.
* **Code Repository Alternative:** If a live interactive endpoint is not feasible, a link to a **public code repository (e.g., GitHub)** is strictly required. This must include comprehensive, detailed setup instructions.

Submissions must explicitly showcase both the project's value and technical implementation. To be eligible, you **must demonstrate at least three (3) of the key course concepts** within your project.

### Core Course Concepts Matrix
| Key Concept | Where to Demonstrate It | Description / Framework Reference |

| **MCP Server** | Code | Model Context Protocol server implementation |

---

### Scoring Breakdown (100 Points Total)

#### Category 1: The Pitch — Problem, Solution, Value (30 Points Total)
Evaluates the "why" and "what" of your project, alongside overall communication clarity.
* **Core Concept & Value (10 Points):** Focuses on the central idea, relevance to the chosen track, innovation, and value. The use of agents must be clear, meaningful, and central to the overall solution.
* **YouTube Video Submission (10 Points):** Evaluates clarity, conciseness, and messaging quality within the strict 5-minute limit (covering Problem, Agent Choice, Architecture, Demo, and Build).
* **Writeup (10 Points):** Assesses how comprehensively and professionally the written submission details the problem, solution, architecture, and project journey.

#### Category 2: The Implementation — Architecture, Code (70 Points Total)
Evaluates the "how" of your project, focusing on technical quality, software design, and AI framework execution.
* **Technical Implementation (50 Points):** Assesses architecture quality, code cleanliness, and meaningful integration of agent technologies. Judges will heavily review tool usage, particularly clever implementation of existing toolsets. 
  * *Code Requirements:* Code must feature descriptive comments regarding implementation, design choices, and agent behaviors.
  * *Deployment Note:* Live deployment is not strictly required, but if deployed, instructions to reproduce it must be clearly documented.
  * *🚨 CRITICAL SECURITY REMINDER:* Do NOT include any API keys, passwords, or credentials in your public codebase.
* **Documentation (20 Points):** For GitHub/repository submissions, a polished `README.md` is required. It must thoroughly explain the problem, solution, system architecture, setup instructions, and relevant architectural diagrams/images.

## 6. Project Strategy Note: Hybrid Integration (Buiry + Google ADK)

*The following notes outline the architectural strategy discussed for integrating an application project (codenamed "Buiry") into the Google Hackathon parameters.*

### The Core Challenge
Because this hackathon evaluates specific course concepts, submitting a purely custom/independent agent stack (e.g., standard Node.js, Anthropic SDK, or non-Google stacks) risks disqualification by the judges who are explicitly looking for the application of course materials (Google ADK, Agents CLI, etc.).

### Selected Strategy: Option B (Hybrid Architecture Layer)
Instead of replacing the core engine of Buiry, a **Hybrid Approach (Option B)** will be pursued to maintain the original product vision while fulfilling 100% of the hackathon criteria.

* **The Concept:** Build a focused Google ADK multi-agent system that sits *on top* of Buiry's existing infrastructure, using Buiry as a persistent memory and context layer via an MCP server interface.
* **How It Maps to Criteria (3 Required Concepts Hit):**
  1. **Agent / Multi-agent system (ADK):** Python-based Google ADK acts as the orchestration layer running the demo agents.
  2. **MCP Server:** The agent calls custom tools (`buiry_start_session`, `buiry_get_context`, `buiry_end_session`) for semantic recall.
  3. **Security Features:** Demonstrable via Buiry's privacy-preserving local data capture pipeline.

### Integration Details & Tooling
* **Confirmed Tech Stack:** Google ADK environment, `google-agent-cli`, Gemini API / Vertex AI, and course-relevant Google tooling.
* **Product Scope Priorities:** Both core user journeys remain first-class features:
  1. The developer using the MCP server for project build memory.
  2. The developer integrating the SDK into their AI product for context/dataset generation.
* **Demo Scenario Plan:** To be detailed later; will showcase the system working natively in action, highlighting all completed features and integration pipelines.
* **Development Phases:** Full chronological execution will be laid out in the updated Product Requirements Document (PRD) and Architecture Document without calendar constraints to prioritize end-to-end flow delivery.
